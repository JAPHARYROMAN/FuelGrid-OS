## 1. Executive framing

### What the Reports & Intelligence Center is

A premium analytics and reporting module that turns fuel-operations data (sales, stock, cash, deliveries, credit, fleet, risk, profitability) into beautiful, interactive, **auditable, drillable, and automated** reports for station managers, finance, auditors, regional managers, and executives — with drill-down traceability from an executive summary down to a source record (blueprint §26).

### The no-AI principle (non-negotiable)

All "intelligence" is generated by **deterministic system logic** — rules, thresholds, period-over-period comparisons, forecasting formulas, data-quality checks, and scheduled jobs — **not an AI assistant** (blueprint §26). This is already how the codebase works: insights come from pure threshold functions in `internal/reporting/` and a configurable-but-deterministic evaluator engine in `internal/risk/`. There is no LLM in the reporting path and none will be added.

### What is genuinely NEW (the product layer above the spine)

The build gap is **persistence + product**, not the data plumbing:

- **§22 report-metadata tables** — `report_templates`, `report_runs`, `report_categories`, `report_snapshots`, `report_subscriptions`, `report_data_quality_warnings`, `report_rules` — **none exist** (categories are a hardcoded Go slice at `reports_structured_handlers.go:92`).
- **Per-tenant DB-driven scheduling** with multi-channel delivery (in-app/email/webhook), recipients-from-DB, per-report cron, permission-at-delivery re-check — **absent** (only two global env digests).
- **Custom Report Builder** (data-source registry + saved views + share/schedule) — **absent entirely**.
- **Report locking / immutable snapshots / revision history / sign-off** — **absent** (closest is `revenue_days.status=locked`, read as a DQ flag).
- **Async export worker** — **absent** (`export_jobs` is a synchronous receipt).
- **Unifying the hardcoded `reporting` composers with the `risk_rules` engine** into one config-driven rules surface.
- **Visualization gap** — no heatmap, donut/pie, stacked bar, financial/P&L waterfall, credit-limit meter, aging-bucket chart, supplier scorecard, forecast bands, status boards; shared `VarianceBadge`/`RiskScoreBadge`/`MoneyField`/`ReportDateRangeFilterBar` do not exist (and there is **no free date-range picker** — filtering is station + 4 fixed period presets).
- **Premium Reports Home** — current hub is a live-metric grid of **8 of 16** categories with no search, no global date/region selector, no hero/alerts band, no recent/scheduled/locked/exports rails.

> **Brief-claim corrections (code is source of truth):** (1) The "Phase 7 attendance + corrections-variances reports" **do not exist** — no such handlers; that data lives in operational reconciliation surfaces. (2) `ReportDateRangeFilterBar` **does not exist**; the filter bar is `ReportFilterBar` (station + period preset only). (3) `EnvelopeTable`/`EnvelopeExports` **do** exist (`report-envelope.tsx`). This plan treats the first two as net-new, not reusable.

### Design-philosophy bar

Premium cockpit, drillable, data-quality-aware. Every report: KPI hero -> report-specific visual -> deterministic insights -> drillable table -> drilldown links + permission-gated export, with a data-quality banner on **every** report. Sensitive figures (margins, supplier cost, credit exposure) are permission-gated. Money and litres are **exact decimal strings end-to-end** (`numeric::text`), never float.

## 3. Cross-cutting decisions to flag for the owner

1. **ClickHouse / analytics-layer decision — defer; Postgres carries the early phases.**
   No ClickHouse or materialized views exist anywhere in code (confirmed) — the substitute is the `revenue_days` rollup (`migrations/0036`, `internal/revenue/days.go`) plus on-read SQL. **Phases 1-10 run entirely on Postgres** and need no new infra. The analytics-layer question only becomes pressing for **Phase 11 (Custom Report Builder)** — arbitrary user-defined queries over large history — and for high-cardinality time-series visuals (sales-by-hour heatmaps over long ranges). **Recommendation:** start now on Postgres; revisit ClickHouse (or Postgres partitioning + continuous aggregates) as an explicit gate **before Phase 11**, driven by measured query latency, not speculatively.

2. **Charting / visualization library — recharts is in; the gap is missing chart *types*, not the library.**
   `recharts ^2.15.4` is installed with token-themed wrappers (`Line/Area/Bar/CategoricalBar/Sparkline`) and signature fuel visuals (`ReconciliationWaterfall`, `TankVisual`, `ShiftTimeline`, `PumpCard`) already exist. **Missing:** heatmap, donut/pie, stacked bar (`BarChart` has no `stackId`), financial/P&L waterfall, credit-limit meter, aging-bucket chart, supplier scorecard, forecast bands, status boards — plus shared `VarianceBadge`, `RiskScoreBadge`, `MoneyField`, `ReportDateRangeFilterBar`. **Recommendation:** **stay on recharts** (donut/pie/stacked are native to it; heatmap and financial waterfall are custom-SVG, same approach as the existing CSS waterfall). Build the missing primitives into `@fuelgrid/ui` **incrementally, owned by the phase that first needs each** (e.g. donut in Phase 2/3, heatmap in Phase 3, financial waterfall in Phase 8). **No new charting dependency required.**

3. **Data-MISSING categories — blocked until the domain exists; do not promise them.**
   - **Tank Sensor / ATG telemetry (§5.4):** only **manual** `water_mm`/`temperature_c` on per-shift dips (`tank_dip_readings` 0017) — no sensor/ATG feed, no time-series table. Tank *water/temperature from manual dips* is buildable; a live **"Tank Sensor Report"** is **MISSING/blocked** until an ingestion domain ships.
   - **Fleet (§5.10):** rich data exists (`customer_vehicles/drivers/fuel_credentials` 0053, `fuel_limits/authorizations` 0054, `vehicle_odometer_readings` 0055; `internal/fleet/odometer.go FleetConsumption`), but there is **no report endpoint** and **"fleet" = a credit customer vehicles fueling on credit, not an internal motor pool** (vehicles hang off `customer_id`). Scope the report accordingly; consumption-per-100km/abuse detection is codeable from existing inputs (odometer distance + litres).
   - These categories should ship as **"data-quality aware" placeholder cards** on the hub (clearly marked unavailable) rather than be faked.

4. **Custom Report Builder (§6) — highest scope/risk; isolate and gate.**
   No data-source registry, no saved-view schema, no builder exist; every report today is a bespoke Go handler. Risks: **query-safety** (arbitrary user filters/joins over tenant data must stay RLS-scoped and permission-aware — margins/cost/credit gating must survive the builder), **performance** (the analytics-layer question above), and **scope creep**. **Recommendation:** constrain v1 to a **whitelisted data-source registry** (curated, permission-tagged views — no free SQL), saved views in `report_templates`, render through the existing `ReportEnvelope`. Treat it as its own L-effort phase **after** the signature reports prove the registry shape.

## 5. Build conventions (recap)

- **Worktree-per-slice, PR-per-slice.** Each phase (and sub-deliverable where large) is its own isolated worktree and its own PR. Never `git add -A` — stage explicit paths only.
- **No AI.** Intelligence is deterministic only — rules, thresholds, period-over-period comparisons, forecasting formulas, data-quality checks, scheduled jobs. Reuse `internal/reporting/` composers and the `risk_rules` evaluator engine; never add an LLM to the reporting path.
- **Decimal strings for money/litres** end-to-end (`numeric::text`), parsed to float only for chart geometry. Never store or transport money as float.
- **Permission-aware + sensitive-data protection.** Every endpoint goes through `requirePermissionHeld` / `resolveStationScoped`; margins, supplier cost, and credit exposure are gated and must stay gated through any builder or export path.
- **RLS + audit/outbox on every write path**; `report.*` audit events via `audit.WriteWithOutbox`.
- **Data-quality on every report** — a DQ banner is mandatory on each report and on the hub; MISSING-data categories render honestly as unavailable.
- **Reuse the existing infra** — `ReportEnvelope`, the export pipeline (CSV/`fpdf`/`excelize`), the advisory-locked scheduler, the outbox + in-app notification substrate. Build the product layer **above** the spine, not beside it.
- **Reserved files untouched** — migrations 0096/0097 are reserved; the current head is 0104, so new report tables begin at **0105+**. Leave `main` alone; work only in worktrees.
